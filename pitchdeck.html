The Challenge

A good database that
could be great

  • 1 The app works today. Living With Fire serves 1,361 plants with 94,903 attribute values from 103 sources — but there's vastly more data sitting in 37 external databases that could enrich every plant profile.
  • 2 Enhancement data can't be safely ingested. The 37 source databases use different schemas, scales, and methodologies. Merging blindly would introduce contradictions and degrade quality.
  • 3 Internal conflicts already exist. The same plant may have contradictory ratings from different sources already in production, with no systematic way to detect or resolve them.
  • 4 Manual curation doesn't scale. 866K records across 37 datasets with overlapping species — a botanist would need years to reconcile by hand.

Why It Matters

In 2020, Oregon wildfires burned 1.2 million acres and destroyed 4,000+ homes. California averages 7,000+ wildfires per year.

Living With Fire already helps homeowners choose fire-safe plants. Enhancing it with deeper, multi-source evidence means better recommendations backed by more science.

Innovation

The Claim/Warrant Model

Inspired by argumentation theory. Every production value is a Claim backed by Warrants — evidence from specific sources with full provenance. Conflicting evidence isn't discarded; it adds nuance.

Warrants (Evidence)

FIRE-01 "Firewise (1)" — literature-based rating
FIRE-02 "Highly resistant, plant 30ft from structures" — practitioner guide
FIRE-04 "Moderate flammability in controlled burn" — lab testing
FIRE-07 "Fire-resistant in 3+ sources" — meta-analysis
AI Synthesized Claim: "Fire-resistant (high confidence, 4 sources). Lab testing showed moderate heat release under controlled conditions, but field sources consistently rate favorably for landscape use. Plant 30ft from structures per Idaho guidelines."

Why This Matters for LWF

The production database already has values from 103 sources. Instead of overwriting them, the Claim/Warrant model enriches existing data — adding depth, nuance, and provenance to every plant profile.

Full Provenance Chain

Every word in a synthesized claim traces back to specific warrants → specific sources → specific files, rows, and methodologies. The data steward can always ask: "Where did this come from?"

Human-in-the-Loop

AI synthesizes; humans approve. The data steward selects which warrants to include, edits AI output, and every decision is logged with full audit trail.

Architecture

Multi-Agent Pipeline

🔍

Match Agent

Resolves species names across taxonomy backbones (POWO, World Flora Online, USDA). Handles synonyms, name changes, and fuzzy matching at genus level.

🛠️

Mapping Agent

Reads each source's DATA-DICTIONARY.md and auto-maps columns to the 125-attribute production schema. Handles rating scale conversion.

⚖️

Conflict Detector

Scans for 8 conflict types: direct contradictions, scale mismatches, scope differences, temporal drift, and more. Prioritizes by severity.

📚

Research Agent

RAG over 52 indexed research documents. When sources disagree, explains why — different methodology, region, time period, or definition.

🤖

Specialist Agents

Domain-specific agents for fire, deer, water, pollinators, and invasiveness. Each understands its field's rating systems and methodologies.

🔬

Synthesis Agent

Combines selected warrants into a single, rich claim. Preserves nuance from conflicting evidence. Produces natural-language explanations.

Built on Firebase Genkit — 12 flows, 13 reusable tools, orchestrated through a Next.js admin portal.
Technology

Built for data integrity

Version-Controlled Staging

DoltgreSQL — Git for databases. Every change is a commit with branching, diffing, blame, and revert. Zero risk to production: the public app never touches the staging layer.

Live Production Database

Neon PostgreSQL — the existing LWF database we're enhancing. 1,361 plants, 125 attributes, 94,903 values, 103 sources. Already serving the public app via REST API on Vercel.

AI Pipeline

Firebase Genkit with the Anthropic Claude API. 12 flows for matching, mapping, conflict detection, research, and synthesis. Cost-optimized: Haiku for bulk matching, Sonnet for synthesis.

Admin Portal

Next.js 16 with App Router. Upload workflows, curation interfaces, conflict queues, coverage dashboards, and sync-to-production. Built for the data steward, not the developer.
